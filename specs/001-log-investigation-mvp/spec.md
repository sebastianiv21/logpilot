# Feature Specification: Log Investigation Platform MVP

**Created**: 2025-03-13  
**Status**: Draft

## Clarifications

### Session 2025-03-13

- Q: How should the system infer service (and optionally environment) when archive structures vary (e.g. path-based like logs/backend/ vs flat or AWS ECS / Cloud Run / ACI)? → A: Derive from archive path when possible; support a small set of well-documented structure conventions for path-based layouts (e.g. logs/<service>/...); other layouts (flat, cloud-provider exports) follow documented rules or fallback to a single upload-scoped label.
- Q: Maximum supported archive size for MVP uploads? → A: Documented hard limit (e.g. 500 MB uncompressed or 100 MB compressed); reject with clear error when exceeded.
- Q: Which files in the archive are treated as log files? → A: Only files matching documented patterns; patterns include extensions .log, .csv, .json (and optionally .log.* or names like stdout/stderr); other entries are skipped and skipped file count is reported in the upload result.
- Q: Multiple uploads: additive or replace? → A: Session-scoped: uploads within a session (e.g. support ticket or investigation) are additive; user can start a new session for a new ticket so data is kept separate. History of sessions is supported; queries and dashboards are scoped to the current or selected session.
- Q: Should incident reports generated by the AI be stored per session for later review and comparison? → A: Yes: store each generated report in the session so the user can review report history and compare reports for that ticket.
- Session/ticket link: Sessions SHOULD support an optional field to link to an external resource (e.g. Slack thread, Zendesk ticket, or any URL) so the user can associate the investigation with the ticket in another platform.
- Session output format: The output for a session (incident report) MUST be exportable as either a Markdown or PDF file. The report MUST include possible root causes, next troubleshooting steps, and everything needed to resolve the issue.
- Report generation and export: Report generation is user-triggered (e.g. the user asks an incident question or explicitly requests "Generate report"); the system does not auto-generate reports. When exporting, the user selects which report to export (e.g. the latest or a specific report from report history), then selects the output format (Markdown or PDF); the exported file contains that single report.
- Session identification: The user MAY set an optional name or label for a session (e.g. ticket ID or description). The session list MUST identify each session by at least creation time and/or external link; if the user set a name/label, it is shown in the list.
- Archive size limits: The values 500 MB uncompressed and 100 MB compressed are the defined MVP limits; the system MUST document and enforce these limits unless the implementation plan explicitly defines and documents different values.
- Knowledge base ingestion: Ingestion is repeatable; the user can trigger ingestion (or re-ingestion) when documentation or repository content is added or changed, so the index stays up to date.
- Environment label: When the archive structure has no environment segment (e.g. path has only service), the environment label is empty or unknown and is optional in queries; users can filter by service (and other labels) without requiring environment.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Ingest and Query Logs (Priority: P1)

As an engineer, I want to upload a compressed archive of application logs so that I can investigate an incident without manually reformatting files. The system extracts and parses log files, detects common formats (plain text, JSON, application log lines), normalizes records with labels (e.g. service, environment, log level), and stores them in a queryable log store. I can then search and filter logs by labels and time range.

**Why this priority**: Without log ingestion and queryability, no other capability (metrics, dashboards, AI) can deliver value. This is the foundation.

**Independent Test**: Can be fully tested by uploading a valid compressed log archive, verifying parsing summary (files processed, lines parsed/rejected), and then running queries by label and time range to retrieve relevant log lines. Delivers value as a standalone log search experience.

**Acceptance Scenarios**:

1. **Given** a valid compressed archive containing one or more log files and a current session (new or existing), **When** the user uploads it, **Then** the system extracts files, parses log lines, normalizes them with required labels, and makes them queryable by label and time range within that session; if the user starts a new session first, the upload applies to the new session and does not mix with other sessions.
2. **Given** an invalid or malformed archive, **When** the user attempts upload, **Then** the system rejects it with a clear, structured error message and does not affect existing data.
3. **Given** mixed parseable and unparseable lines in a file, **When** the system processes the upload, **Then** parseable lines are ingested; unparseable lines are tracked and reported in the parsing summary without failing the whole upload.
4. **Given** ingested logs in a session, **When** the user queries by service, environment, log level, or time range (within the current or selected session), **Then** the system returns matching log records with preserved raw message content for evidence review.
5. **Given** an existing session with prior uploads, **When** the user uploads another archive to the same session, **Then** the new batch is additive; all data in the session remains queryable. **Given** the user starts a new session for a new ticket, **When** they upload an archive, **Then** that data is stored in the new session only and is separate from other sessions; the user can switch between sessions and see session history.
6. **Given** a session (new or existing), **When** the user sets an optional external link (e.g. Slack thread, Zendesk ticket, or any URL), **Then** the link is stored with the session and is visible when viewing or listing sessions so the investigation can be associated with the ticket in another platform.
7. **Given** a session, **When** the user sets an optional name or label (e.g. ticket ID or description), **Then** the name/label is stored and shown in the session list; the list identifies each session by at least creation time and/or external link (and name if set).

### User Story 2 - View Derived Metrics and Dashboards (Priority: P2)

As an engineer, I want metrics derived from my uploaded logs (e.g. error counts, request volume, error rate, and when available response time distribution) so that I can spot spikes, anomalies, and failure windows. I want these metrics visible in pre-provisioned dashboards so I can inspect system health immediately after upload without manual configuration.

**Why this priority**: Observability value depends on moving from raw logs to visual trends; metrics and dashboards are the next step after logs are stored.

**Independent Test**: Can be fully tested by ingesting logs that contain the necessary fields, verifying that derived metrics are available for query and visualization, and confirming that at least one default dashboard shows error rate, request volume, error distribution, log volume, and anomalies over time. Delivers value as a standalone observability view.

**Acceptance Scenarios**:

1. **Given** ingested logs, **When** the system has processed them, **Then** derived metrics (e.g. errors total, requests total, error rate, and response time distribution when present) are available for query and time-window analysis.
2. **Given** a fresh local setup, **When** the user starts the platform, **Then** at least one default dashboard is available and shows metric panels and log exploration for uploaded data.
3. **Given** dashboard views, **When** the user inspects a time period, **Then** they can correlate metric panels with log exploration for the same period.
4. **Given** logs that lack fields required for a specific metric, **When** the system generates metrics, **Then** unsupported metrics are skipped with clear status feedback and do not break the pipeline.

---

### User Story 3 - Search Documentation and Repository Context (Priority: P3)

As an engineer, I want the system to index documentation, runbooks, and repository content into a searchable knowledge base so that incident investigation can use historical knowledge and code context. Content should be chunked, retrievable by semantic similarity, and include source metadata (e.g. file path, document type).

**Why this priority**: AI investigation requires retrievable knowledge; indexing must be in place before the agent can cite docs and code.

**Independent Test**: Can be fully tested by ingesting a set of markdown/text and source files, running semantic search queries, and verifying that relevant chunks are returned with source references and metadata. Delivers value as a standalone documentation/code search experience.

**Acceptance Scenarios**:

1. **Given** documentation, runbooks, or repository files (e.g. markdown, text, source code), **When** the user triggers ingestion (or re-ingestion after adding or changing content), **Then** the system chunks content, generates embeddings, stores them in a searchable index with source metadata; ingestion is repeatable so the user can refresh the index when docs or repo content changes.
2. **Given** an indexed knowledge base, **When** a search query is executed, **Then** the system returns relevant chunks with source references and metadata (e.g. file path, repository, document type).
3. **Given** retrieved results, **When** the user reviews them, **Then** they can trace evidence back to the original document or code location.

---

### User Story 4 - AI-Assisted Incident Investigation and Structured Report (Priority: P4)

As an engineer, I want to ask a natural-language incident question (e.g. "Why did the service fail yesterday?") and receive a structured incident report that cites evidence from logs, metrics, documentation, and repository context. The report must include possible root causes, next troubleshooting steps, and everything needed to resolve the issue. I want to export the session output as a Markdown or PDF file to share or attach (e.g. to a support ticket). The system must use only approved tools (query logs, query metrics, search docs, search repo, generate report) and produce a consistent structure: Incident Summary, Possible Root Cause, Supporting Evidence, Recommended Fix, next troubleshooting steps. When evidence is insufficient, the system must state uncertainty clearly.

**Why this priority**: This is the main differentiator and user-facing outcome, but it depends on P1–P3 being available.

**Independent Test**: Can be fully tested by asking at least one end-to-end incident question after logs, metrics, and knowledge are available; verifying that the agent uses the approved tools; and confirming the final report includes the required sections and references logs, metrics, and at least one knowledge source where relevant. Delivers value as a single end-to-end investigation flow.

**Acceptance Scenarios**:

1. **Given** ingested logs, derived metrics, and an indexed knowledge base, **When** the user triggers report generation (e.g. by asking an incident question or requesting "Generate report"), **Then** the agent uses only approved tools and returns a structured report with Incident Summary, Possible Root Cause, Supporting Evidence, Recommended Fix, and next troubleshooting steps (and everything needed to resolve the issue); the report is stored in the session. **Given** one or more generated reports in a session, **When** the user exports and selects which report to export (e.g. latest or a specific report from history) and then selects format (Markdown or PDF), **Then** the user obtains that single report in the chosen format.
2. **Given** an incident question, **When** the agent has sufficient evidence, **Then** the final answer references logs, metrics, and at least one knowledge source (docs or repo) where relevant.
3. **Given** weak or insufficient evidence, **When** the agent cannot form a confident conclusion, **Then** the report states uncertainty explicitly and does not present unsupported claims as fact.
4. **Given** retrieved documentation or repository content, **When** it is used in the agent workflow, **Then** it cannot override system instructions (prompt injection resistance); tool inputs and outputs are validated.
5. **Given** a session with one or more previously generated reports, **When** the user views that session, **Then** they can access report history and compare reports for that ticket.

### Edge Cases

- What happens when the user uploads an empty archive or an archive with no recognizable log files? The system should reject or report clearly (e.g. zero files processed) and not fail silently.
- What happens when all lines in a file are unparseable? The system should report the count of rejected lines and still complete the upload; no logs are ingested for that file but the pipeline remains stable.
- What happens when the knowledge base is empty and the user asks an incident question? The agent should still use logs and metrics and explicitly indicate when documentation or repo context was not available or not relevant.
- What happens when the user asks a question outside the available time range of logs or metrics? The agent should scope queries to available data and state time-range or data limitations in the report.
- What happens when upload extraction encounters path traversal or unsafe paths? The system must sanitize filenames and restrict filesystem access so that extraction cannot write outside the intended directory.
- What happens when a chosen metric is not derivable from the uploaded logs (e.g. no latency field)? The system skips that metric and provides status feedback rather than failing.
- What happens when the archive layout does not match any documented structure convention (e.g. unknown cloud export)? The system applies the fallback: use a single upload-scoped label (e.g. one logical service per upload) so logs remain queryable; the upload result and documentation MUST make the supported conventions and fallback behavior clear.
- When the archive structure has no environment segment (e.g. path has only service), the environment label is empty/unknown; queries can filter by service (and other labels) without requiring environment.
- What happens when the archive exceeds the documented maximum size (e.g. 500 MB uncompressed or 100 MB compressed)? The system rejects the upload with a clear, structured error message and does not process or extract the archive.
- What happens when the archive contains no files matching the log-file patterns (e.g. only .txt or binary files)? The system completes extraction, reports zero files processed and the number of files skipped, and does not ingest any log lines; the upload result makes the outcome clear.
- How does the user add logs to the same ticket vs start a new ticket? The user adds logs to the current session (additive); to start a new ticket they start a new session, so data is kept per session and session history allows switching between tickets.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST accept upload of compressed log archives (e.g. zip) and extract one or more log files from the archive. Only entries matching documented file patterns MUST be processed as log files; patterns MUST include at least extensions .log, .csv, .json and MAY include .log.* or common names (e.g. stdout, stderr). Other archive entries MUST be skipped; the upload result MUST include the number of files skipped (or equivalent) so users understand what was processed.
- **FR-002**: The system MUST support common log formats including plain text logs, JSON logs, and typical application log lines, and MUST reject or handle unsupported or malformed archives gracefully with a structured error message. The system MUST enforce a documented maximum archive size; for MVP the limits are 500 MB uncompressed and 100 MB compressed (the implementation plan MUST document these or explicitly define and document different values). Uploads that exceed the limit MUST be rejected with a clear, structured error message.
- **FR-003**: The system MUST parse, normalize, and enrich log lines with at least: timestamp (where possible), log level, and service name. Service (and environment when present) MUST be derived from archive structure using documented conventions: path-based layouts (e.g. a top-level segment like `logs/<service>/...`) are the primary supported convention; flat or other structures (e.g. from AWS ECS, Google Cloud Run, Azure Container Instances) MUST follow documented mapping rules or fallback to a single upload-scoped label. When the archive structure has no environment segment, the environment label MUST be empty or unknown and MUST be optional in queries (users can filter by service without requiring environment). The system MUST produce a canonical internal schema with labels suitable for querying (e.g. service, environment, log_level).
- **FR-004**: The system MUST store normalized logs in a queryable store supporting filter by label and time range, and MUST preserve raw log message content for evidence review. Logs and derived metrics MUST be scoped by session (e.g. support ticket or investigation); within a session, uploads are additive (multiple batches). The system MUST allow the user to start a new session for a new ticket so that data from different tickets is kept separate. The system MUST support a notion of current or selected session for queries and dashboards and MUST support session history (list or switch between sessions). The session list MUST identify each session by at least creation time and/or external link. The user MAY set an optional name or label for a session (e.g. ticket ID or description); if set, it MUST be shown in the session list. The system MUST allow the user to associate a session with an optional external link (e.g. Slack thread, Zendesk ticket, or any URL) so the investigation can be linked to the ticket in another platform; the link is stored with the session and visible when viewing or listing sessions.
- **FR-005**: The system MUST return an upload result that includes at least: upload status, number of files processed as logs, number of files skipped (not matching log-file patterns), lines parsed, and lines rejected.
- **FR-006**: The system MUST derive from ingested logs at least: errors total, requests total, error rate, and when latency is present in logs, response time distribution; and MUST expose these for visualization and time-window analysis.
- **FR-007**: The system MUST provision at least one default dashboard so that users can inspect metrics and log volume and correlate metric panels with log exploration for the same period without manual setup. Dashboard data MUST be scoped to the current or selected session.
- **FR-008**: The system MUST support ingestion of documentation, runbooks, and repository content (e.g. markdown, text, source code), chunk content into retrievable segments, and store them in a searchable index with source metadata (e.g. file path, document type). Ingestion MUST be repeatable: the user can trigger ingestion (or re-ingestion) when docs or repo content is added or changed, so the index can be refreshed.
- **FR-009**: The system MUST support a configurable model provider (e.g. via environment or configuration) so that users can choose their LLM backend without code or deployment changes.
- **FR-010**: The system MUST provide an AI agent that accepts natural-language incident questions and uses only approved tools: query logs, query metrics, search documentation, search repository, and generate incident report. The agent MUST operate in the context of the current or selected session (logs and metrics scoped to that session); knowledge base (docs, repo) may be shared across sessions.
- **FR-011**: The system MUST produce structured incident reports containing: Incident Summary, Possible Root Cause, Supporting Evidence, Recommended Fix, next troubleshooting steps, and everything needed to resolve the issue. Report generation MUST be user-triggered (the user asks an incident question or explicitly requests report generation; the system does not auto-generate reports). Reports MUST be exportable from the session; when exporting, the user MUST select which report to export (e.g. the latest or a specific report from report history), then select the output format (Markdown or PDF); the exported file contains that single report. The system MUST preserve intermediate evidence for auditability and MUST store each generated report in the session so the user can review report history and compare reports for that ticket.
- **FR-012**: The system MUST validate tool inputs and outputs, apply time-range and query-scope controls, and ensure retrieved content cannot override system instructions (prompt injection resistance).
- **FR-013**: The system MUST run locally via a single orchestration workflow so that the full stack (upload, log store, metrics, dashboards, knowledge index, AI agent) can be started and used for investigation without mandatory cloud or third-party data egress for core flows.
- **FR-014**: The system MUST limit AI tools to read-only investigation actions in MVP (no production remediation or arbitrary execution); MUST sanitize uploaded filenames and extracted archive paths and restrict filesystem access during extraction.

### Key Entities

- **Session**: A logical container for an investigation or support ticket. Each session has its own logs and derived metrics; uploads within a session are additive (multiple batches). User can start a new session for a new ticket; queries and dashboards are scoped to the current or selected session. Session history is maintained so the user can switch or refer back to previous sessions; the list identifies each session by at least creation time and/or external link. A session MAY have an optional name or label (e.g. ticket ID or description) and an optional external link (e.g. Slack thread, Zendesk ticket, or any URL); if set, name and link are stored and shown when viewing or listing sessions.
- **Log archive**: A compressed bundle (e.g. zip) containing one or more log files; input to the upload flow for a given session. Archive structure may vary (e.g. path-based such as `logs/<service>/...`, or flat/cloud-provider exports); service and environment labels are derived per documented structure conventions, with fallback to a single upload-scoped label when the structure is unknown.
- **Parsed log record**: A normalized log entry with timestamp, level, service, labels, raw message, and optional metadata (source filename, batch, session).
- **Derived metric**: A numeric measure computed from log events (e.g. errors total, requests total, error rate, response time distribution) for a time window, used for visualization and analysis.
- **Dashboard**: A pre-provisioned set of views showing metrics and log exploration for uploaded data, correlatable by time range.
- **Knowledge chunk**: A segment of documentation or repository content with embeddings and source metadata, stored in a searchable index for retrieval by semantic similarity.
- **Incident report**: Structured output from the AI agent containing Incident Summary, Possible Root Cause, Supporting Evidence, Recommended Fix, next troubleshooting steps, and everything needed to resolve the issue; with references to logs, metrics, and knowledge sources. Reports are stored per session so the user can review history and compare reports. When exporting, the user selects which report to export (e.g. latest or one from history), then the format (Markdown or PDF); the exported file contains that single report.

## Assumptions

- Users provide log archives manually for MVP; no automated cluster-wide or Kubernetes log collection.
- Repository and documentation sources are available locally or via mounted paths; no mandatory live sync or GitHub OAuth in MVP.
- Incident investigation is retrospective (user-initiated questions) rather than real-time or alert-triggered in the initial release.
- Users supply their own API keys for their chosen model provider; the platform does not bundle or mandate a single provider.
- Initial users are technical and comfortable with API-driven workflows and dashboard-based inspection.
- Maximum archive size is a documented hard limit: for MVP, 500 MB uncompressed and 100 MB compressed are the defined limits; the system documents and enforces these (or the implementation plan defines and documents different values). Repository indexing method (manual vs mounted vs Git-driven) may be documented or constrained in setup guides; reasonable defaults apply for MVP.
- Archive structures vary (path-based e.g. logs/<service>/..., flat, or cloud-provider exports such as ECS, Cloud Run, ACI); the system documents supported structure conventions and uses a single upload-scoped label when the layout does not match.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A user can upload a compressed log archive and view parsed, queryable logs in the platform (by label and time range) without manual reformatting.
- **SC-002**: The system generates and stores derived metrics from uploaded logs; these metrics are visible and queryable for visualization and time-window analysis.
- **SC-003**: A fresh local setup includes working default dashboards; users can inspect metrics and pivot to logs for the same period without manual configuration.
- **SC-004**: Relevant documentation and code snippets can be retrieved from the knowledge base for incident questions, with source references and metadata.
- **SC-005**: The AI agent answers at least one end-to-end incident question using only approved tools; the final report includes summary, likely cause, supporting evidence, recommended fix, and next troubleshooting steps (everything needed to resolve the issue), and references logs, metrics, and at least one knowledge source when relevant. The user can select which report to export (from report history), then export it as a Markdown or PDF file.
- **SC-006**: The full stack runs locally via a single orchestration workflow; core flows (upload, query, metrics, dashboards, knowledge retrieval, AI report) do not require mandatory cloud or third-party data egress.
- **SC-007**: Upload-to-query latency is short enough for interactive use after ingestion completes; partial or batch failures do not crash the system.
- **SC-008**: When evidence is insufficient, the agent states uncertainty explicitly; tool calls are validated and retrieved content cannot override system instructions.
